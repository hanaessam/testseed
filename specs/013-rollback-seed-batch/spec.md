# Feature Specification: Rollback Seed Batch

**Feature Branch**: `015-rollback-seed-batch`

**Created**: 2026-06-09

**Status**: Draft

**Input**: User description: "Epic #17 Rollback [core]. As a developer, I want to rollback a seed batch. The user can select or enter a seedBatchId. The system deletes records from the batch and reports deleted counts by collection. Acceptance Criteria: The user can provide a seedBatchId. The system validates that the batch exists. Only records created by that seedBatchId are deleted. Records from other seed batches are never deleted. The rollback report includes deleted counts by collection. Collections are processed in reverse dependency order. If the batch ID is missing, an error is returned. If the batch ID is invalid, an error is returned. If the batch was already rolled back, an error is returned. If no records exist for the batch, an error is returned. The system must not delete unrelated data. MongoDB connections are closed after the operation. Scope: Core-only implementation. MongoDB native driver. Use the seedBatchId generated by Direct MongoDB Seeding. Rollback operates only on records tagged with the specified seedBatchId. Preserve dependency safety by deleting collections in reverse generation order. Return a structured rollback report. Do not implement UI changes in this epic. Do not implement rollback history persistence beyond what is required to prevent duplicate rollback execution."

## Clarifications

### Session 2026-06-09

- Q: How should the system detect an already rolled back seed batch without adding broader rollback history persistence? -> A: Update existing seed batch metadata with rollback status and deleted counts; no separate rollback history collection.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Roll Back a Seed Batch (Priority: P1)

As a developer, I can provide a seedBatchId for a previous direct seeding operation so records from that batch are deleted and I receive deleted counts by collection.

**Why this priority**: This is the primary user value: developers need a safe way to undo a direct seed operation without manually deleting generated records.

**Independent Test**: Can be tested by preparing records tagged with one seedBatchId, running rollback for that batch, and confirming only matching records are deleted while a structured report lists deleted counts.

**Acceptance Scenarios**:

1. **Given** records exist for a provided seedBatchId, **When** the developer requests rollback, **Then** the system deletes records tagged with that seedBatchId.
2. **Given** rollback completes, **When** the report is returned, **Then** it includes the seedBatchId and deleted counts by collection.
3. **Given** records from other seed batches exist in the same collections, **When** rollback runs, **Then** those other records are not deleted.

---

### User Story 2 - Preserve Dependency Safety During Rollback (Priority: P1)

As a developer rolling back relational seed data, I want collections processed in reverse dependency order so dependent records are removed before parent records.

**Why this priority**: Reverse dependency order reduces the risk of temporarily leaving child records that reference deleted parent records during rollback.

**Independent Test**: Can be tested with a multi-collection batch whose original generation order is parent-first and confirming rollback deletes collections in the reverse order.

**Acceptance Scenarios**:

1. **Given** a seed batch includes parent and dependent collections, **When** rollback runs, **Then** dependent collections are processed before parent collections.
2. **Given** generation order is available for the batch, **When** rollback order is determined, **Then** rollback uses the reverse of that generation order.

---

### User Story 3 - Reject Unsafe or Duplicate Rollback (Priority: P1)

As a developer, I want missing, invalid, already rolled back, or empty seed batches rejected so rollback never deletes unrelated data or repeats a completed rollback.

**Why this priority**: Rollback is destructive. Guardrails are required to prevent accidental deletion and duplicate rollback attempts.

**Independent Test**: Can be tested by requesting rollback with missing, invalid, already rolled back, and no-record batch states and confirming each returns an error without deleting records.

**Acceptance Scenarios**:

1. **Given** no seedBatchId is provided, **When** rollback is requested, **Then** an error is returned and no records are deleted.
2. **Given** an invalid seedBatchId is provided, **When** rollback is requested, **Then** an error is returned and no records are deleted.
3. **Given** the seed batch was already rolled back, **When** rollback is requested again, **Then** an error is returned and no records are deleted.
4. **Given** no records exist for the seedBatchId, **When** rollback is requested, **Then** an error is returned and no records are deleted.

---

### Edge Cases

- If the seedBatchId is missing, empty, or whitespace, rollback is rejected before any delete operation.
- If the seedBatchId does not match the expected seed batch identifier format, rollback is rejected before any delete operation.
- If the batch cannot be validated as existing, rollback is rejected before any delete operation.
- If the batch is already marked or known as rolled back, rollback is rejected before any delete operation.
- If no records tagged with the seedBatchId are found in target collections, rollback is rejected and reports that no batch records exist.
- If some target collections contain records from other seed batches, those records are not deleted.
- If rollback encounters a collection deletion failure after earlier collections succeeded, the report distinguishes completed deletions from failed collections.
- If rollback succeeds, fails, or is rejected after opening a connection, the database client connection is closed.
- Connection strings must remain transient and must never appear in reports, errors, logs, persisted history, or saved datasets.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST allow a developer to provide a seedBatchId for rollback.
- **FR-002**: The system MUST reject rollback when seedBatchId is missing, empty, or whitespace.
- **FR-003**: The system MUST reject rollback when seedBatchId is invalid.
- **FR-004**: The system MUST validate that the seed batch exists before deleting records.
- **FR-005**: The system MUST reject rollback when the seed batch was already rolled back.
- **FR-006**: The system MUST reject rollback when no records exist for the seedBatchId.
- **FR-007**: The system MUST delete only records tagged with the specified seedBatchId.
- **FR-008**: The system MUST NOT delete records tagged with other seedBatchIds or records without the specified seedBatchId.
- **FR-009**: The system MUST process collections in reverse generation order.
- **FR-010**: The rollback report MUST include the seedBatchId and deleted counts by collection.
- **FR-011**: If rollback partially fails, the system MUST stop on the first collection deletion failure and the report MUST identify collections that completed and the collection that failed.
- **FR-012**: The system MUST update existing seed batch metadata with rollback status and deleted counts to prevent duplicate rollback execution for the same seedBatchId, without creating a separate rollback history collection.
- **FR-013**: The system MUST close database client connections after rollback operations, including success, failure, and rejection paths that open a connection.
- **FR-014**: This feature MUST keep rollback business logic in core and MUST NOT add UI changes; any API work MUST be adapter-only.
- **FR-015**: This feature MUST NOT add rollback history persistence beyond what is required to prevent duplicate rollback execution.
- **FR-016**: Rollback MUST use the seedBatchId created by direct MongoDB seeding and operate only on records tagged with that seedBatchId.
- **FR-017**: Rollback MUST use the approved MongoDB connection mechanism for this epic while keeping connection strings transient.

### Key Entities

- **Rollback Request**: The developer's active rollback operation containing a seedBatchId, target collections, generation order context, and transient connection information.
- **Seed Batch**: A prior direct seeding operation identified by a seedBatchId and collection/order metadata needed to locate and safely delete inserted records.
- **Rollback Status**: The state stored on existing seed batch metadata to determine whether the batch is eligible for rollback or was already rolled back.
- **Rollback Report**: The structured result containing seedBatchId, deleted counts by collection, completed collections, failed collections, and safe error summaries.
- **Rollback Collection Result**: Per-collection deletion outcome with collection name, deleted count, and failure details when applicable.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% of rollback attempts without a seedBatchId are rejected before deleting records.
- **SC-002**: 100% of rollback attempts with invalid seedBatchIds are rejected before deleting records.
- **SC-003**: 100% of rollback attempts for already rolled back batches are rejected before deleting records.
- **SC-004**: 100% of rollback attempts with no records for the seedBatchId are rejected before deleting records.
- **SC-005**: 100% of successful rollback operations delete only records tagged with the specified seedBatchId.
- **SC-006**: 0 records from other seed batches are deleted in rollback verification.
- **SC-007**: 100% of multi-collection rollback operations process collections in reverse generation order.
- **SC-008**: 100% of rollback reports include seedBatchId and deleted counts by collection.
- **SC-009**: 100% of tested success, failure, and applicable rejection paths close the database client connection after the operation.
- **SC-010**: No connection string appears in rollback reports, errors, logs, persisted history, or saved datasets during verification.

## Assumptions

- The rollback caller can supply the same target database context used for direct seeding without storing connection strings.
- The seedBatchId format follows the direct seeding batch identifier format.
- The rollback caller can supply or resolve the generation order for the batch; rollback uses the reverse of that order.
- Existing seed batch metadata may be updated with minimal rollback status and deleted counts only to prevent duplicate rollback execution for a seedBatchId.
- Rollback does not restore deleted records, undo non-TestSeed writes, or delete records that lack the specified seedBatchId.
- Web UI changes, long-term rollback history, and saved connection-string behavior are outside this feature. API changes are limited to adapter-only request validation, transient MongoDB connection handling, and delegation to the core rollback use case.
